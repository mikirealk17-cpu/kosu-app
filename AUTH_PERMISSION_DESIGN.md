# ログイン・権限管理 設計メモ

この文書は、将来ログイン機能を追加して、工数データ・単価・請求金額を安全に扱うための設計メモです。

2026-07-22時点では、第1段階として `作業者` と `システム管理者` の2権限を実装しています。
`会社管理者` は将来の複数会社運用に備えた設計案として残しています。

## 目的

- 管理者以外が他人の工数や金額を見られないようにする
- 作業者本人は自分の工数だけ確認できるようにする
- 会社管理者は自社に関係する作業者の工数だけ確認できるようにする
- 画面で隠すだけでなく、Supabase側でもデータ取得を制限する

## 権限の種類

### システム管理者

アプリ全体を管理する人です。

できること:

- 全会社の工数を閲覧
- 全会社のExcel出力
- 作業者、元請け、製番、作業内容、単価の管理
- 請求確認CSVの出力

想定:

- アプリ所有者、または全体管理者

### 会社管理者

自分の会社に所属する作業者を管理する人です。

できること:

- 自社に所属する作業者の工数を閲覧
- 自社に所属する作業者のExcel出力
- 自社に関係する請求確認

できないこと:

- 他社の作業者の工数閲覧
- 他社の請求金額閲覧
- 全体マスタの無制限編集

### 作業者

実際に工数を入力する人です。

できること:

- 自分の工数入力
- 自分の工数履歴確認
- 自分の分だけ履歴確認

できないこと:

- 他人の工数閲覧
- 他人の請求金額閲覧
- 単価マスタの閲覧・編集

## 追加するテーブル案

### company_master

作業者が所属する会社を管理します。

```text
id
name
is_active
created_at
updated_at
```

### user_profiles

Supabase Authのログインユーザーと、アプリ内の作業者・会社・権限を紐づけます。

```text
id
auth_user_id
worker_id
company_id
role
is_active
created_at
updated_at
```

role は以下の3種類にします。

```text
system_admin
company_admin
worker
```

## 既存テーブルに追加する列案

### worker_master

作業者がどの会社に所属するかを持たせます。

```text
company_id
```

### work_logs

検索とRLSを安定させるため、保存時点の所属会社IDを持たせる案です。

```text
company_id
```

理由:

- 作業者の所属会社が後で変わっても、過去工数の所属を変えないため
- Excel出力時に会社単位の絞り込みを安全にするため

## RLS方針

画面側で非表示にするだけでは不十分です。
必ずSupabaseのRLSで、DB側でも読める行を制限します。

基本ルール:

- system_admin は全行を閲覧可能
- company_admin は自分の company_id と一致する行だけ閲覧可能
- worker は自分の worker_id と一致する行だけ閲覧可能

対象テーブル:

- work_logs
- worker_master
- rate_master
- billing_company_master
- seiban_master
- work_type_master

金額に関わる `rate_master` と `billing_amount` は特に慎重に扱います。

## 画面方針

### ログイン前

- ログイン画面だけ表示
- 工数入力、集計、管理画面には入れない

### system_admin

- すべての画面を表示
- 管理画面も表示

### company_admin

- 工数入力
- 集計画面
- 履歴編集
- 自社分のExcel出力
- 必要最小限の管理画面

### worker

- 工数入力
- 自分の履歴確認
- 自分の履歴確認
- 単価管理や全体集計は非表示

## 実装順

安全のため、以下の順で進めます。

1. 設計書を作る
2. Supabase Authを使う方針を確定する
3. `company_master` と `user_profiles` のSQLを作る
4. 既存データへ company_id を付与する手順を作る
5. ログイン画面を追加する
6. ログイン中ユーザーのプロフィール取得を追加する
7. 画面表示を権限ごとに切り替える
8. RLSを読み取り専用から段階的に追加する
9. Excel出力を権限ごとに絞り込む
10. テストユーザーで確認する

2026-07-22時点の実装状況:

- `login.html` と `login.js` を追加
- 共通認証ガード `auth.js` を追加
- 作業者は工数入力と本人履歴だけ利用可能
- 管理者は集計、Excel出力、履歴、各マスタ管理を利用可能
- 第1段階では作業者プロフィールの `company_id` は任意で、`worker_id` を必須にする
- RLS本体は `SUPABASE_AUTH_RLS_POLICIES.sql` として分離し、ログイン確認後に実行する

## テスト方針

最低限、以下を確認します。

- 作業者Aでログインすると、作業者Aの工数だけ見える
- 作業者Aでログインしても、作業者Bの工数は見えない
- 会社管理者でログインすると、自社作業者の工数だけ見える
- 会社管理者でログインしても、他社作業者の工数は見えない
- システム管理者でログインすると、全データが見える
- Excel出力も画面表示と同じ範囲に制限される
- 単価と金額が不要な権限には表示されない

## 注意点

- 既存データは自動削除しない
- 既存の工数データに company_id を付ける時は、必ずバックアップ後に行う
- RLSを一気に厳しくすると画面が全く表示できなくなる可能性があるため、段階的に有効化する
- 認証実装後は、公開用anonキーだけでは全データを読めない状態にする
